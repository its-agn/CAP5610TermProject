\section{Problem Setup}
\label{sec:prob-setup}

\subsection{Classification Task}

The task is five-class single-label text classification. Given the raw text of a Yelp review, the model must predict the star rating the reviewer assigned, which is one of five ordinal classes ranging from 1 star to 5 stars. Each review belongs to exactly one class. The ordinal structure of the labels means that misclassifying a 1-star review as 2-star is a smaller error than misclassifying it as 5-star, though all standard metrics used here treat all misclassifications equally.

\subsection{Dataset}

The dataset used is \textbf{Yelp Review Full}, a large-scale sentiment classification benchmark compiled from the 2015 Yelp Dataset Challenge. It consists of English-language Yelp business reviews paired with their star ratings. Reviews vary considerably in length, from a single sentence to several paragraphs, and cover a wide range of business categories. The dataset is publicly available through HuggingFace Datasets under the identifier \texttt{yelp\_review\_full}.

\subsection{Input and Output Format}

\textbf{Input:} Raw English review text as a variable-length string. No manual preprocessing is required before passing to the data loader. Sparse feature models (Naive Bayes, MLP, SVM, Logistic Regression, Decision Tree, Random Forest) convert this text to a fixed-size TF-IDF vector via the shared \texttt{fit\_tfidf\_features()} utility. Sequence models tokenize and index the text into fixed-length integer sequences.

\textbf{Output:} A single integer class prediction in the range $\{0, 1, 2, 3, 4\}$, corresponding to star ratings 1 through 5. Labels are zero-indexed internally throughout the codebase; the mapping to human-readable star ratings is applied only at evaluation and display time.

\subsection{Basic Dataset Statistics}

The dataset is approximately balanced across all five classes. Each class accounts for exactly 20\% of both the training and test partitions, with minor variation. Training class counts are approximately 130,000 reviews per class before the validation split is carved out. This near-uniform distribution means accuracy is a meaningful primary metric and no class weighting or oversampling is required.

Reviews vary substantially in length. Short reviews of one to two sentences are common in the extreme rating classes (1 star and 5 stars), where the reviewer's sentiment is unambiguous and concisely expressed. Middle-rating reviews (2, 3, and 4 stars) tend to be longer and more qualified, mixing positive and negative observations, which contributes to the difficulty of separating these classes.

\subsection{Project Goals}

The primary goal is to implement and compare ten machine learning models spanning traditional and deep learning approaches on a shared text classification benchmark. Each model is evaluated under a consistent experimental protocol using the same dataset splits, evaluation metrics, and result logging infrastructure. The secondary goal is to quantify the tradeoff between model complexity, input representation quality, and predictive performance, with particular attention to how much improvement sequence-aware deep learning models provide over bag-of-words baselines.

\subsection{Data Preparation and Experimental Pipeline}

\subsubsection{Data Loading Workflow}

All models read the dataset through a single entry point: \texttt{utils.data.load\_yelp\_data()}, which calls \texttt{datasets.load\_dataset("yelp\_review\_full")}. The dataset downloads on first use and is cached locally. Following runs then read from cache.

\subsubsection{Training/Validation/Test Protocol}

The official split of the dataset is 650,000 training reviews and 50,000 test reviews. The loader then takes an additional validation split (by default 10\%) out of the training partition using \texttt{sklearn.model\_selection.train\_test\_split}. The test partition is never touched during tuning. Validation runs use a default 150,000-review subsample of training data for fast iteration; final runs use the full 650,000-review training set (unless specified otherwise for some models) and evaluate on the held-out 50,000-review test partition (triggered by \texttt{--final}).

\subsubsection{Text Preprocessing}

Sparse-feature models (Logistic Regression, Naive Bayes, Decision Tree, Random Forest, Linear/Kernel SVM, MLP) consume TF--IDF features built directly from raw review text by \texttt{TfidfVectorizer}, which lowercases, tokenizes, and applies \texttt{strip\_accents="unicode"} internally. Sequence models (CNN, LSTM, Transformer) lowercase the text and replace punctuation with whitespace before tokenizing on whitespace, then map tokens to fixed-size index sequences (CNN: \texttt{max\_len = 256};  LSTM: \texttt{max\_len = 1024}; Transformer: \texttt{max\_len = 160}). Tokens outside of the vocabulary collapse to \texttt{<unk>} and zero-padded positions are masked at the embedding layer. The decoder-only LLM (default \texttt{distilgpt2}) bypasses this pipeline entirely and instead applies the model's pretrained HuggingFace \texttt{AutoTokenizer} (subword BPE) directly to the raw text, padding/truncating to \texttt{max\_length = 256}.

\subsubsection{Feature Engineering}

\textbf{TF--IDF:} \texttt{fit\_tfidf\_features()} fits a single \texttt{TfidfVectorizer} with \texttt{sublinear\_tf=True} (dampens the contribution of repeated words within a review via $1 + \log(\text{tf})$) and four tunable parameters: \texttt{max\_features} (cap on the vocabulary size), \texttt{ngram\_range} (whether to include bigrams in addition to unigrams), \texttt{min\_df} (drop terms appearing in fewer than this many documents, removing rare/typo tokens), and \texttt{max\_df} (drop terms appearing in more than this fraction of documents, removing extremely common words). Final feature dimensionalities range from 10,000 (MLP) to 40,000 (Decision Tree, Random Forest, Logistic Regression).

\textbf{Word vocabulary:} Each word-level sequence model builds its own top $N$ vocabulary from the training text with index 0 reserved for the padding token (\texttt{<pad>}) and index 1 reserved for the unknown word token (\texttt{<unk>}). CNN: 25,000 words (fixed); LSTM: 30,000 words (tuned over 10k--75k); Transformer: 15,000 words (tuned over 15k--30k). Reviews are truncated or zero-padded to the model's \texttt{max\_len}.

\textbf{Embeddings:} The CNN supports either a randomly initialized 256-dimensional embedding lookup (trained from scratch) or one of several pretrained sources: GloVe 6B 100d/300d, GloVe 42B 300d, GloVe 2024 WikiGigaword 300d, and fastText wiki-news subword 300d, which can all be selected with mutually exclusive CLI flags. Pretrained matrices are loaded into the embedding layer at model construction time. The LSTM uses a trainable \texttt{nn.Embedding} lookup tuned over {128, 256, 300} dimensions with no pretrained option. The Transformer learns both token and positional embeddings from scratch (token embedding tuned over {128, 256} dimensions, positional embedding sized to \texttt{max\_len}). The decoder-only LLM gets its subword embeddings from the pretrained \texttt{distilgpt2} checkpoint and tunes them jointly with the rest of the model.

\subsubsection{Randomness and Reproducibility}

A single global seed (\texttt{DEFAULT\_SEED = 0}) is set at the start of every run via \texttt{set\_random\_seed()}, which seeds all of the following: Python's \texttt{random}, NumPy, and Torch (including CUDA), and pins \texttt{torch.backends.cudnn.deterministic = True} with \texttt{benchmark = False}. The same seed is passed into the dataset splitter and into every Optuna study, so re-running any script reproduces the same train/validation split and the same trial order.

\subsubsection{Hyperparameter Tuning Workflow}

Tuning is invoked with the \texttt{--tune} flag and uses Optuna Bayesian optimization through \texttt{utils.tuning.tune\_model()}. Each model defines an \texttt{objective()} that samples a configuration, builds features, trains on a subset of the training data ranging from roughly 6,000 reviews (LLM) to 650,000 reviews (CNN, Linear SVM) depending on the model's compute cost per trial, and returns macro F1 on the validation split. Trial budgets range from 6 (Logistic Regression) to 30 (CNN), with most models in the 15--20 range. When tuning completes, \texttt{save\_best\_config()} writes the winning configuration plus metadata (seed, embedding source, train size, validation split, trial count, best macro F1) to \texttt{best\_config.json}, and the tuning helper appends a Markdown summary of all trials to \texttt{tuning\_log.md}. Following runs load that JSON automatically unless \texttt{--default} is passed. Naive Bayes is the exception: \texttt{--tune} runs a manual alpha sweep over a fixed grid rather than an Optuna study, and writes \texttt{nb\_alpha\_sweep.png} instead of a tuning log.

\subsection{Traditional Machine Learning Models}

\subsubsection{Logistic Regression}
\textbf{Folder/Script:} \texttt{Logistic\_Regression/Logistic\_Regression.py}\\
\textbf{Team Member:} Mohammed Mamdouh

\paragraph{Model Description:}\mbox{}
A multinomial logistic regression classifier from \texttt{sklearn.linear \_model. LogisticRegression}. The model learns one linear score per class over the TF--IDF feature vector and applies a SoftMax based cross entropy objective for review rating prediction. Unlike Naive Bayes, it does not assume feature independence, but it is still a linear bag of words model and therefore only captures limited local order through the chosen $n$-gram features.


\paragraph{Input Representation:}
Sparse TF--IDF features built directly in the script with \texttt{TfidfVectorizer}. The vectorizer uses \texttt{sublinear\_tf=True}, \texttt{strip\_accents="unicode"}, and \texttt{dtype=np.float32}. The tuned configuration keeps 40,000 features, includes unigrams and bigrams, and sets \texttt{min\_df = 2}, \texttt{max\_df = 0.9}.


\paragraph{Training Procedure:}
Validation runs load a 150,000 review subset and hold out 10\% for validation where final runs fit on the full 650,000-review training split and evaluate on the 50,000 review test split. Optimization is handled by scikit-learn rather than a manual training loop. The script uses \texttt{lbfgs} for \texttt{l2} regularization and switches to \texttt{saga} when \texttt{elasticnet} is selected.


\paragraph{Hyperparameters:}
The tunable parameters and their Optuna search spaces are:

\begin{table}[H]
    \centering
    \begin{tabular}{lll}
        \toprule
        Parameter & Type & Search Space \\
        \midrule
        \texttt{tfidf\_features} & Categorical & \{20000, 40000, 60000\} \\
        \texttt{ngram\_max} & Integer & \{1, 2\} \\
        \texttt{min\_df} & Categorical & \{1, 2, 5\} \\
        \texttt{max\_df} & Categorical & \{0.9, 0.95, 1.0\} \\
        \texttt{C} & Float (log) & $[0.25, 8.0]$ \\
        \texttt{penalty} & Categorical & \{\texttt{l2}, \texttt{elasticnet}\} \\
        \texttt{max\_iter} & Categorical & \{200, 300, 500\} \\
        \texttt{l1\_ratio} & Float & $[0.05, 0.95]$ when \texttt{elasticnet} is used \\
        \bottomrule
    \end{tabular}
\end{table}

The convergence tolerance remains fixed at \texttt{1e-4}.


\paragraph{Tuning Strategy:}
Optuna Bayesian optimization with 6 trials on a 100,000 review training subset and a 10\% validation split. Search space: \texttt{tfidf\_features} $\in \{20000, 40000, 60000\}$, \texttt{ngram\_max} $\in \{1, 2\}$, \texttt{min\_df} $\in \{1, 2, 5\}$, \texttt{max\_df} $\in \{0.9, 0.95, 1.0\}$, \texttt{C} $\in [0.25, 8.0]$ on a log scale, \texttt{penalty} $\in \{\texttt{l2}, \texttt{elasticnet}\}$, \texttt{max\_iter} $\in \{200, 300, 500\}$, and \texttt{l1\_ratio} $\in [0.05, 0.95]$ when elastic net is active. The saved best configuration uses \texttt{l2} regularization with $C \approx 0.379$ and achieved validation macro F1 of 0.6002 during tuning.


\paragraph{Implementation Notes:}
The full pipeline stays lightweight: TF--IDF fitting, model training, evaluation, and artifact saving all happen in one script while reusing the shared repo utilities for dataset loading, metrics, and logging. Because the representation remains sparse end-to-end and training is CPU-only, Logistic Regression serves as one of the fastest strong baselines in the project. Its final macro precision, macro recall, and macro F1 are all tightly clustered near 0.615, which suggests the model is already making fairly balanced decisions across classes and that most of the remaining limitations comes from the linear TF--IDF representation from being a non deep learning model rather than an easily fixable training issue.


\subsubsection{Naive Bayes}
\textbf{Folder/Script:} \texttt{Naive\_Bayes/naive\_bayes.py}\\
\textbf{Team Member:} Benedetto Falin

\paragraph{Model Description:}
Complement Naive Bayes (CNB) from \texttt{sklearn.naive\_bayes.ComplementNB}. CNB is a variant of Multinomial Naive Bayes that estimates class-conditional probabilities using the \textit{complement} of each class rather than the class itself. For each class $c$, it computes the likelihood of a feature given all other classes combined, which has been shown to produce better-calibrated weight estimates on imbalanced or heavily overlapping text corpora than standard Multinomial NB. The model assumes conditional independence of features given the class label (the Naive Bayes assumption), meaning word co-occurrence and order are not modeled.

\paragraph{Input Representation:}
TF-IDF vectors produced by the shared \texttt{fit\_tfidf\_features()} helper with \texttt{sublinear\_tf=True}, \texttt{max\_features = 20{,}000}, \texttt{ngram\_range = (1, 2)}, \texttt{min\_df = 5}, \texttt{max\_df = 0.9}. Bigrams are included so that two-word phrases such as ``not good'' or ``really great'' are represented as single features, partially compensating for the model's inability to reason about word order at inference time. The resulting sparse matrix is passed directly to the sklearn fit and predict methods with no further transformation.

\paragraph{Training Procedure:}
CNB training is a single-pass frequency count followed by a log-probability computation -- there is no iterative optimization. Final runs fit on the full 585,000-review training partition and evaluate on the 50,000-review test partition. Validation runs use the 150,000-review subsample with a 10\% validation split for the alpha sweep. Training is CPU-only and completes in under 0.2 seconds on the full partition.

\paragraph{Hyperparameters:}
The only tunable model hyperparameter is \texttt{alpha}, the Laplace/Lidstone additive smoothing coefficient. Higher alpha values add more probability mass to unseen feature-class combinations, preventing zero-probability estimates that would dominate the log-likelihood. The four TF-IDF parameters listed above are fixed rather than tuned for this model.

\paragraph{Tuning Strategy:}
Rather than Optuna, a manual grid sweep over a fixed list of alpha values is performed when \texttt{--tune} is passed:
\[
\alpha \in \{0.01,\ 0.05,\ 0.1,\ 0.3,\ 0.5,\ 1.0,\ 2.0,\ 5.0,\ 10.0,\ 15.0,\ 20.0,\ 25.0,\ 30.0,\ 35.0,\ 40.0,\ 45.0,\ 50.0\}
\]
For each candidate, the model is fit on the training subset and validation accuracy is recorded. The alpha achieving the highest validation accuracy is saved to \texttt{nb\_best\_config.json} and used for all subsequent runs. A validation accuracy versus alpha plot is saved as \texttt{nb\_alpha\_sweep.png}. The best alpha found was \textbf{45}. This value is higher than what was observed under the original unigram-only pipeline (where alpha of 30 was optimal), and the increase is attributable to the addition of bigrams in the TF-IDF feature set. Bigram features are considerably more sparse than unigram where a phrase like ``not great'' appears far less frequently than either constituent word, which reduces the average discriminative reliability of individual features and requires heavier smoothing to prevent the model from over-fitting to noisy frequency counts.

\paragraph{Implementation Notes:}
The unusually high optimal alpha is an interpretable finding in itself. Standard smoothing values (0.1 to 1.0) are sufficient when classes are well-separated in feature space. The fact that alpha must reach 45 before validation accuracy peaks indicates that individual word frequencies are weak discriminators across the five classes and heavy smoothing is needed to prevent the model from over-committing to any single term. This is consistent with the known difficulty of the middle star ratings (2, 3, and 4 stars), where reviews share substantial vocabulary regardless of the assigned rating.


\subsubsection{Decision Tree}
\textbf{Folder/Script:} \texttt{Decision\_Tree/Decision\_Tree.py}\\
\textbf{Team Member:} Keating Sane

\textbf{Model Description:} A single decision tree classifier from \texttt{sklearn.tree.DecisionTreeClassifier}. The tree is built top-down by repeatedly choosing one TF--IDF feature and a threshold that best separates the five star-rating classes (using Gini impurity), then recursing on each side. Predictions follow the chain of feature-vs-threshold tests from the root to a leaf, where the majority class is returned. Implemented as a baseline against the bagged ensemble (Random Forest).

\textbf{Input Representation:} TF--IDF vectors produced by the shared \texttt{fit\_tfidf\_features()} helper with \texttt{sublinear\_tf=True}, \texttt{max\_features = 40,000}, \texttt{ngram\_range = (1, 2)}, \texttt{min\_df = 10}, \texttt{max\_df = 0.9}. No additional preprocessing.

\textbf{Training Procedure:} Final runs fit a single tree on the full 650,000-review training partition with the tuned config and evaluate on the 50,000-review test partition. Validation runs use a 150,000-review training subsample with a 10\% validation split for fast iteration. CPU-only.

\textbf{Hyperparameters:} The tunable parameters are \texttt{max\_depth} (maximum tree depth, which caps model capacity and the rate at which the tree can carve up the feature space), \texttt{min\_samples\_leaf} (minimum training rows per leaf, which prevents the tree from creating tiny leaves that memorize individual reviews), and the four TF--IDF parameters listed above. The splitting criterion is sklearn's default Gini impurity.

\textbf{Tuning Strategy:} Optuna Bayesian optimization with 20 trials on a 200,000-review tuning subset (10\% validation split). Search space: \texttt{tfidf\_features} $\in$ \{5k, 10k, 20k, 40k\}, \texttt{ngram\_max} $\in$ \{1, 2\}, \texttt{min\_df} $\in$ \{3, 5, 10\}, \texttt{max\_df} $\in$ \{0.85, 0.9, 0.95\}, \texttt{max\_depth} $\in$ \{50, 100, 150, 300, None\}, \texttt{min\_samples\_leaf} $\in [1, 10]$. macro F1 on the validation split was the objective.

\textbf{Implementation Notes:} Tree fitting on the 200k tuning subset takes well under a minute per trial on CPU, but the full 650k-review final fit grows to roughly an hour as the tree depth scales. With the best tuned settings, the model memorizes most of the training set: train accuracy is far above the macro F1 it achieves on validation/test. The model was kept as an interpretable baseline to compare with the ensemble.

\subsubsection{Random Forest}
\textbf{Folder/Script:} \texttt{Random\_Forest/Random\_Forest.py}\\
\textbf{Team Member:} Keating Sane

\textbf{Model Description:} Ensemble of decision trees from \texttt{sklearn.ensemble.RandomForestClassifier}. Each tree is fit on a bootstrap sample of the training rows with a random subset of features considered at every split (\texttt{max\_features = "sqrt"}); class predictions are averaged across the forest. The randomized feature subsampling reduces the variance that the single Decision Tree exhibits.

\textbf{Input Representation:} TF--IDF vectors from \texttt{fit\_tfidf\_features()} with \texttt{sublinear\_tf=True}, \texttt{max\_features = 40,000}, \texttt{ngram\_range = (1, 2)}, \texttt{min\_df = 10}, \texttt{max\_df = 0.85}. Identical pipeline to the single tree, with a slightly tighter \texttt{max\_df}.

\textbf{Training Procedure:} Final runs fit a 500-tree forest on the full 650,000-review training partition with the tuned config and evaluate on the 50,000-review test partition. Validation runs use a 150,000-review training subsample with a 10\% validation split. Tree fitting is parallelized across cores via \texttt{n\_jobs = -1}; CPU-only.

\textbf{Hyperparameters:} The tunable parameters are mirrored from the Decision Tree, with two additions: \texttt{n\_estimators} (number of trees in the forest) and \texttt{max\_features} (number of features randomly considered at each split).

\textbf{Tuning Strategy:} Optuna Bayesian optimization with 20 trials on a 200,000-review tuning subset (10\% validation split). Search space mirrors the Decision Tree's parameters and adds \texttt{n\_estimators} $\in$ \{100, 200, 300, 500\} and \texttt{max\_features} $\in$ \{"sqrt", "log2"\}. macro F1 on the validation split was the objective.              

\textbf{Implementation Notes:} A 500-tree forest on the 200k tuning subset fits in roughly three minutes per trial with \texttt{n\_jobs = -1}; the full 650k-review final fit grows to about 44 minutes. Bootstrap sampling plus the \texttt{max\_features = "sqrt"} per-split feature subset force each tree to see a different slice of the data and consider a different subset of words at every split, so the trees make different mistakes and the variance that the single Decision Tree exhibits averages out.

\subsubsection{Linear SVM}
\textbf{Folder/Script:} \texttt{Linear\_SVM/Linear\_SVM.py}\\
\textbf{Team Member:} Alexander Nardi

\paragraph{Model Description:} A Linear Support Vector Machine (Linear SVM) from \texttt{sklearn.svm.LinearSVC}. This model learns a set of linear decision boundaries that maximize the margin between the review classes (1-5 stars). Linear SVM's operate directly in the original, high dimensional feature space (in this case, TF-IDF's), which make it extremely efficient for sparse, high-dimensional data. The model optimizes hinge loss with L2 regularization.


\paragraph{Input Representation:} TF--IDF vectors produced by the \texttt{sklearn.feature\_extraction.text. TfidVectorizer} with optimal maximum features found in tuning and english stop-words. The standard TF--IDF's sparse representation is well-suited for Linear SVMs, seeing as the model scales linearly, as the name implies, with the amount of non-zero features.


\paragraph{Training Procedure:} The model is trained using a coordinate descent, implemented in sklearn. Final runs fit on the full training partition and evaluate on the test set. Validation runs for tuning use a subsampled training set with a 10\% validation split for hyperparameter tuning. Training is CPU-only and completes relatively quickly in only about 100 seconds or so.


\paragraph{Hyperparameters:} The primary hyperparameter is \texttt{C}, the inverse regularization strength. Smaller values of \texttt{C} enforce stronger regularization, resulting in a wider margin but resulting in potentially higher bias. Whereas larger values do the opposite, weakening regularization, and allowing the model to fit the training data more closely. The secondary hyperparameter was the maximum length \texttt{max\_features}, which is the number of features considered


\paragraph{Tuning Strategy:} Optuna Bayesian optimization with 15 trials on the maximum 650,000-review training set, with a 10\% validation split. The full training set is allowed as training is relatively quick, and this allows us to find optimal hyperparameters for the ones listed above. \texttt{C} is selected as the best float between \{0.1, 10.0\}, whereas \texttt{max\_features} is selected between \{5000, 10,000, \& 15,000\}. The optimal hyperparameters is noted to be \texttt{C = 0.10619348907224617} and \texttt{max\_features = 15,000}. Maximizing the macro F1 score was used as the objective for hyperparameter tuning.


\paragraph{Implementation Notes:} Linear SVM was actually particularly well-suited for this task due to the high dimensionality and the inherent sparsity in a TF--IDF Bag-of-Words approach. Even then, the model performed rather shockingly high on a NLP task, getting around 57\% accuracy and surpassing models which should theoretically have performed better as Linear SVM cannot process word order.

\subsubsection{Kernel SVM}
\textbf{Folder/Script:} \texttt{Kernel\_SVM/Kernel\_SVM.py}\\
\textbf{Team Member:} Anthony Mahon
 
\paragraph{Model Description:}
A kernel Support Vector Machine using \texttt{sklearn.svm.SVC} with a Radial Basis Function (RBF) kernel. SVMs learn a decision boundary that maximizes the margin between the nearest training points (support vectors) of each class. The soft-margin objective allows some misclassifications, controlled by the penalty parameter $C$.

The kernel trick replaces the explicit mapping $\phi$ with a kernel function evaluated on pairs of input vectors, enabling the model to find non-linear boundaries without ever computing the high-dimensional feature map. The RBF kernel measures similarity between two TF--IDF vectors.

\paragraph{Input Representation:}
TF--IDF vectors produced by \texttt{sklearn.feature\_extraction.text. TfidfVectorizer} with \texttt{stop\_words='english'} and a vocabulary capped at \texttt{max\_features}. TF--IDF assigns higher weight to words that are frequent within a specific review but rare across the full corpus (e.g., ``undercooked'' or ``phenomenal''), making it a strong signal for sentiment-loaded vocabulary. No additional normalization is applied beyond what \texttt{TfidfVectorizer} performs internally (\texttt{norm='l2'} by default).
 
\paragraph{Training Procedure:}
Exact kernel SVM training requires computing and storing the full $n\times n$ kernel matrix, giving $O(n^2)$ memory and $O(n^3)$ time complexity. Training on the full 650,000-review corpus is therefore infeasible. The model is trained on a stratified subsample of 10,000 reviews by default (up to 20,000 with \texttt{--size N}). A 20\% stratified validation split is carved from this subsample for tuning; final runs evaluate on the official 50,000-review test partition. Training is CPU-only.
 
\paragraph{Hyperparameters:}
Two parameters were tuned:
\begin{itemize}
    \item \texttt{C}: soft-margin penalty. Smaller values allow more misclassifications but produce wider, more generalizable margins; larger values force a tighter fit to the training data.
    \item \texttt{max\_features}: vocabulary cap for the TF--IDF vectorizer. Controls the size of the input space presented to the kernel.
\end{itemize}
 
\paragraph{Tuning Strategy:}
Optuna Bayesian optimization with 15 trials on a 10,000-sample subsample (20\% validation split). Search space: $C \in [0.1, 10.0]$ (log scale), \texttt{max\_features} $\in \{5000, 10000, 15000\}$. Macro-F1 on the validation split was the objective. Notably, the best Optuna configuration did not outperform a manual setting of $C=1$ (the sklearn default), which was ultimately selected. This is expected behavior: with only 10,000 training samples the validation signal is noisy, and the well-validated default $C=1$ generalizes more reliably than configurations tuned on a small subsample. Best config: $C=1$, \texttt{max\_features }$=10{,}000$.
 
\paragraph{Implementation Notes:}
The training-set size cap is the dominant limitation of this model. The linear SVM and logistic regression train on 65 times more data and comfortably outperform the kernel SVM as a consequence. The RBF kernel's theoretical advantage (non-linear boundaries) cannot compensate for this data deficit. Theoretically, the model should be among the top performers, but in practice, it is difficult to use its structure within reasonable computation times.

\subsubsection{Multi-Layer Perceptron}
\textbf{Folder/Script:} \texttt{MultiLayer\_Perceptron/ml\_perceptron.py}\\
\textbf{Team Member:} Benedetto Falin

\paragraph{Model Description:}
A fully connected feedforward neural network implemented in PyTorch (\texttt{torch.nn.Sequential}). The MLP takes a fixed-size TF-IDF vector as input and learns a non-linear mapping to five class logits via two hidden layers with batch normalization, ReLU activations, and dropout regularization. Unlike Naive Bayes, the MLP can learn interactions between input features, but like all TF-IDF-based models it has no access to word order or sequential context. Trained end-to-end with cross-entropy loss via backpropagation.

\paragraph{Input Representation:}
TF-IDF vectors from the shared \texttt{fit\_tfidf\_features()} helper with \texttt{sublinear\_tf=True}, \texttt{max\_features = 10{,}000}, \texttt{ngram\_range = (1, 2)}, \texttt{min\_df = 5}, \texttt{max\_df = 0.9}. The sparse matrix is converted to a dense \texttt{float32} PyTorch tensor in chunks of 10,000 rows to avoid attempting a full dense allocation of approximately 20 GB. The resulting dense tensor is wrapped in a \texttt{TensorDataset} and served through a \texttt{DataLoader} with \texttt{pin\_memory=True} for efficient GPU transfer.

\paragraph{Architecture:}
\[

No softmax is applied at the output; \texttt{nn.CrossEntropyLoss} applies the log-softmax internally. The default architecture uses $h_1 = 1024$, $h_2 = 512$.

\paragraph{Training Procedure:}
Optimizer is Adam with a \texttt{ReduceLROnPlateau} scheduler that halves the learning rate when validation accuracy does not improve for 2 consecutive epochs. Early stopping with patience of 3 halts training when validation accuracy has not improved for 3 consecutive epochs, and the best-validation-accuracy checkpoint is restored before evaluation. Final runs train on the full 585,000-review training partition for up to 15 epochs and evaluate on the 50,000-review test partition. Validation runs use the 150,000-review subsample.

\subsubsection{Transformer (Encoder Only)}
\textbf{Folder/Script:} \texttt{Transformers/Transformers.py}\\
\textbf{Team Member:} Anthony Mahon
 
\paragraph{Model Description:}
An encoder-only Transformer trained entirely from scratch on the YelpReviewFull dataset, implemented in PyTorch. The model follows the standard Transformer encoder design introduced by Vaswani et al.\ (2017): stacked multi-head self-attention blocks allow every token in a review to attend to every other token simultaneously, capturing long-range dependencies that recurrent models process sequentially. Unlike BERT or other pre-trained models, no external linguistic knowledge is used; every parameter, including the word embeddings, is initialized randomly and learned solely from the training reviews. This means that the model is not only learning the structure of the reviews, but also the structure of the English language itself.
 
\paragraph{Input Representation:}
Raw review text is lowercased and stripped of punctuation via a regular expression filter (\texttt{r"[\^{}\textbackslash w\textbackslash s]"}). The resulting token stream is split on whitespace and mapped to integer IDs using a frequency-ranked vocabulary built from the training corpus, capped at \texttt{max\_vocab=30,000} entries. Words outside the top 30k most common collapse to \texttt{<unk>} (index 1); sequences are truncated to a fixed length \texttt{max\_len}, producing an integer matrix of shape $[\texttt{batch\_size} \times \texttt{max\_len}]$.
 
\paragraph{Architecture:}
The model is implemented as \texttt{EncoderClassifier} in \texttt{Transformers / Transformers.py}, and consists of four components:
\begin{enumerate}
    \item \textbf{Token embedding} (\texttt{nn.Embedding}): maps each token ID to a dense \texttt{embed\_dim}-dimensional vector. Padding positions use \texttt{padding\_idx=0} and are excluded from attention via a key-padding mask passed to the encoder.
    \item \textbf{Positional embedding} (\texttt{nn.Embedding}): a second learnable embedding maps each position index $(0, 1, \ldots, \texttt{max\_len}-1)$ to an \texttt{embed\_dim}-dimensional vector. Token and position embeddings are summed element-wise, giving the model awareness of word order.
    \item \textbf{Transformer encoder stack} (\texttt{nn.TransformerEncoder}): \texttt{num\_layers} \texttt{TransformerEncoderLayer} blocks, each containing multi-head self-attention followed by a position-wise feed-forward sub-network with GELU activation. Self-attention is computed as:
    \[
    \text{Attn}(\mathbf{Q},\mathbf{K},\mathbf{V}) =
    \text{softmax}\!\left(\frac{\mathbf{Q}\mathbf{K}^\top}{\sqrt{d_k}}\right)\mathbf{V},
    \quad d_k = \frac{\texttt{embed\_dim}}{\texttt{num\_heads}}.
    \]
    The \texttt{batch\_first=True} convention is used throughout.
    \item \textbf{Classification head}: non-padding token representations are mean-pooled (masking padding positions to prevent them from influencing the average), passed through layer normalization and dropout, and projected to five logits by a linear layer. The highest logit selects the predicted star rating.
\end{enumerate}
 
\paragraph{Training Procedure:}
The optimizer used is AdamW (\texttt{torch.optim.AdamW}) with cross-entropy loss over the five-class softmax output. Training runs for \texttt{epochs} full passes over the training set in mini-batches of size \texttt{batch\_size}. After each epoch, macro-F1 is computed on the validation split and the best-scoring model state is saved separately; the best state is restored before final evaluation. Final runs use the full 650,000-review training partition (minus a 5\% stratified validation slice to identify the best model state) and require a GPU; the final reported run used a Tesla T4 provided through Google Collab.
 
\paragraph{Hyperparameters:}
Ten Optuna trials were run on a 20,000-sample training subsample (10\% validation split). The search space was:

        \toprule
        Parameter & Type & Search Space \\
        \midrule
        \texttt{max\_vocab} & Categorical & \{15000, 25000, 30000\} \\
        \texttt{max\_len}   & Categorical & \{96, 128, 160\} \\
        \texttt{embed\_dim} & Categorical & \{128, 256\} \\
        \texttt{num\_heads} & Categorical & \{4, 8\} \\
        \texttt{num\_layers}& Categorical & \{2, 3\} \\
        \texttt{ff\_dim}    & Categorical & \{256, 512\} \\
        \texttt{dropout}    & Float       & $[0.1,\ 0.3]$ \\

Optuna Bayesian optimization via \texttt{utils.tuning.tune\_model()}, objective is macro-F1 on the validation split. The tuning subsample (20,000 reviews) was chosen as the largest size that balances computation time on a T4, allowing 10 meaningful trials within a reasonable time budget.

\paragraph{Implementation Notes:}
The implementation pretokenizes the dataset into a small custom \texttt{Dataset} object and uses \texttt{DataCollatorWithPadding} so each batch is padded only to its local maximum sequence length. On CUDA, training uses automatic mixed precision through \texttt{torch.autocast} and \texttt{GradScaler} to reduce memory pressure. The best model state is tracked in memory using validation macro F1, restored after training, and also written to \texttt{Large\_Language\_Model/saved\_model} so the exact fine-tuned checkpoint can be reused later.


\subsection{Cross-Model Engineering Choices}

\subsubsection{Common CLI Design}

Every model script imports \texttt{common\_parser()} from \texttt{utils.args}, which registers four shared flags: \texttt{--tune} (run hyperparameter search), \texttt{--final} (full training set, evaluate on the test partition), \texttt{--discard} (run without writing any artifacts), and \texttt{--default} (ignore any saved \texttt{best\_config.json} and use hard-coded defaults). Model-specific flags (e.g.\ the CNN's embedding-source selectors, the SVMs' \texttt{--size}, the LLM's checkpoint and resume flags) are layered on top of this shared parser. \texttt{--tune} is mutually exclusive with \texttt{--final} and \texttt{--default}.

\subsubsection{Logging and Artifact Saving}

Run output is structured by helpers in \texttt{utils.evaluation}: \texttt{print\_run\_header()} prints the model name and contextual info; \texttt{print\_value\_section()} prints the parameter table; \texttt{timed\_step()} prints elapsed time for each step during loading/training. After evaluation, \texttt{save\_results()} appends a Markdown section to \texttt{results\_log.md} with accuracy, macro precision, macro recall, macro F1, runtime, parameters, and metadata; the same helper guards against overwriting a better run by comparing macro F1 against the existing entry. For final runs, \texttt{plot\_confusion\_matrix()} writes a PNG figure into the model folder. \texttt{--discard} bypasses all writes.

\subsubsection{Hardware/Runtime Considerations}

Sparse feature models run on CPU and parallelize through \texttt{n\_jobs=-1} where supported. The CNN, LSTM, Transformer, MLP, and LLM auto-select MPS, CUDA, or CPU at startup and use a fixed size token batch loader. Final run hardware varied: NVIDIA RTX 5090 (Linear SVM, CNN), AMD Ryzen 7 9800X3D (Decision Tree, Random Forest), AMD Ryzen 7 7800X3D (Naive Bayes, MLP), Tesla T4 (Transformer), and Intel i7-1165G7 (Kernel SVM). Final runtimes are recorded per-run in each \texttt{results\_log.md}.

\section{Evaluation Metrics}

\subsection{Notation}

Let $N$ be the number of evaluated reviews and $K = 5$ the number of classes. For review $i$, let $y_i \in \{1,\dots,5\}$ be the true label and $\hat{y}_i$ the predicted label. For each class $c$, let $TP_c$, $FP_c$, and $FN_c$ denote the usual true positive, false positive, and false negative counts under a view of that class. These are the same quantities used by the shared \texttt{compute\_metrics()} helper in \texttt{utils/evaluation.py}.


\subsection{Accuracy}
\textbf{Formula:}
\[
    \mathrm{Accuracy} = \frac{1}{N}\sum_{i=1}^{N} \mathbf{1}[y_i = \hat{y}_i]
\]
\textbf{Interpretation:}
Accuracy is the fraction of reviews assigned the correct star rating. It is the easiest metric to read, but by itself it does not show whether a model is doing equally well across all five classes.

\subsection{Macro Precision}
\textbf{Formula:}
\[
    \mathrm{MacroPrecision} = \frac{1}{K}\sum_{c=1}^{K} \frac{TP_c}{TP_c + FP_c}
\]
\textbf{Interpretation:}
Precision for one class asks: when the model predicts class $c$, how often is it right? Macro precision averages that quantity equally over all five star ratings, so strong performance on the easy classes cannot fully hide weak precision on the harder middle ratings, which for our dataset rang especially true for differentiating 2-4 star ratings.

\subsection{Macro Recall}
\textbf{Formula:}
\[
    \mathrm{MacroRecall} = \frac{1}{K}\sum_{c=1}^{K} \frac{TP_c}{TP_c + FN_c}
\]
\textbf{Interpretation:}
Recall for one class asks: of the reviews that truly belong to class $c$, how many did the model recover? Macro recall gives every class the same weight, which makes it useful for checking whether the classifier is systematically missing certain star ratings.

\subsection{Macro F1 Score}
\textbf{Formula:}
\[
    F1_c = \frac{2P_cR_c}{P_c + R_c},
    \qquad
    \mathrm{MacroF1} = \frac{1}{K}\sum_{c=1}^{K} F1_c
\]
\textbf{Interpretation:}
Macro F1 combines precision and recall for each class, then averages the five class wise F1 values. This is the main score used for tuning in the repository because it rewards models that are consistently good across the label set rather than models that inflate one metric at the expense of the other.

\subsection{Confusion Matrix}
\textbf{Definition/Formula:}
\[
    M_{ij} = \sum_{n=1}^{N} \mathbf{1}[y_n = i \ \land\ \hat{y}_n = j]
\]
\textbf{Interpretation:}
Entry $M_{ij}$ counts how many reviews from true class $i$ were predicted as class $j$. In the saved figures, rows correspond to the true labels and columns correspond to the predicted labels. Unlike the scalar metrics above, the confusion matrix shows exactly where the errors occur, such as confusion between adjacent ratings like 3-star and 4-star reviews.

\subsection{Why These Metrics Fit This Project}

This project is a five balanced class text classification problem, so a single metric is not enough. Accuracy gives a clean overall summary, but it can hide whether the model mainly succeeds on the easy extreme ratings while struggling on the ambiguous middle ratings. Macro precision, macro recall, and macro F1 fix that by giving each class equal weight in the final score. That matters here because the practical difficulty is not just getting many predictions right; it is separating five closely related sentiment levels.

The repository therefore logs all four scalar metrics on every run and uses macro F1 as the tuning objective. That choice is reasonable for this dataset as again, the labels are balanced, but the classes are not equally easy where differentiating 0 and 5 star reviews is much simpler than 2 and 3 star reviews. The confusion matrix then provides the missing detail by showing which star levels are being mixed up. Together, these metrics give both a concise summary and an error breakdown that is specific enough to compare very different model families fairly.

This confirms that pre-trained contextual representations from GPT-style language modeling capture the nuanced mixed-sentiment signal that is the primary source of difficulty in this task. Adjacent-class confusion still dominates the off-diagonal entries: 2-star predicted as 1-star (2,047) is the largest single error, and 4-star predicted as 5-star (2,345) is the next largest. Long-range errors (e.g., 1-star predicted as 5-star: only 61 cases) are negligibly rare, confirming all models capture broad sentiment direction. The main cost of the LLM's top performance is runtime: at 17,006 seconds (283 minutes) on an RTX 5070 Ti, it is approximately 75 times slower than the TextCNN while gaining only 2.9 points of macro-F1.

\subsection{Hyperparameter Sensitivity}
Selected examples of sensitivity observed in tuning:
\begin{itemize}[leftmargin=1.5em, itemsep=0.1em]
    \item \textbf{Linear SVM:} performance was only weakly sensitive to $C$ inside the explored range; the selected $C\approx0.11$ regularizes more aggressively than the default.
    \item \textbf{Decision Tree:} tree depth alone did not solve the sparse-feature mismatch; leaf-size regularization was necessary for stable validation performance.
    \item \textbf{Random Forest:} increasing to hundreds of trees improved over the default, with \texttt{max\_features}$=$\, sqrt selected by tuning.
    \item \textbf{LSTM:} sequence length was the most impactful parameter; tuning selected \texttt{max\_len}$=1024$, far exceeding initial defaults, capturing full review context that shorter windows miss. Embedding dimension (300 vs.\ 128) and bidirectionality both produced consistent gains; dropout was less sensitive in the range explored.
    \item \textbf{MLP:} final results were sensitive to optimization settings; the tuned configuration improved over logistic regression but only by a small margin.
    \item \textbf{CNN:} the embedding source was the largest practical knob; GloVe 2024 WikiGigaword 300d gave the best measured final result.
    \item \textbf{Transformer:} limited training budget and from-scratch embeddings capped performance relative to pre-trained-representation approaches.
    \item \textbf{LLM:} learning rate was the most sensitive parameter; the best trial used $1.28\times10^{-5}$, a conservative value appropriate for fine-tuning a pre-trained model without catastrophic forgetting. Sequence length of 256 outperformed 128 and 192, and 2 epochs outperformed 1, suggesting the model benefited from a second full pass without overfitting.
\end{itemize}
 
\subsubsection{Performance vs.\ Compute Tradeoffs}
 
Complement Naive Bayes completes in under one second and achieves 0.5372 macro-F1. Logistic Regression takes roughly 17 minutes and reaches 0.6156. The TextCNN takes 226 seconds on an RTX 5090 GPU and achieves 0.6408. The LSTM takes 9,894 seconds (165 minutes) on an RTX 4080 Super and achieves 0.6687. The fine-tuned LLM achieves the best result of 0.6701 but requires 17,006 seconds (283 minutes) on an RTX 5070 Ti---roughly 75 times slower than the TextCNN for a gain of only 2.9 macro-F1 points. The Transformer takes 1,296 seconds on a Tesla T4 and achieves only 0.5847, making it the worst compute-per-accuracy trade-off among the deep models. The practical takeaway is that the LSTM offers the best accuracy-per-compute balance among sequence models: it closes most of the gap to the LLM at less than 60\% of the training time, while the TextCNN remains the most efficient option overall if pre-trained sequential context is not required.
 
\subsubsection{Error Patterns}
 
Across all models, the dominant error pattern is adjacent-class confusion. Reviews at the boundary between 2 and 3 stars, or between 3 and 4 stars, share heavily overlapping vocabulary and are systematically harder than 1-star or 5-star reviews for every model tested. The 3-star class consistently has the lowest per-class recall across all models, as mixed-sentiment reviews can plausibly belong to multiple adjacent classes. Long-range errors (predicting 5 stars for a true 1-star review) are rare and occur less than 1\% of the time in the better models, confirming that all models have captured at least a broad notion of sentiment direction even when they fail on fine-grained distinctions.

\section{Conclusions \& Future Work}

\subsection{Key Findings}
Across models, performance metrics (accuracy, precision, recall, and macro F1) tended to nearly match each other for each individual model. This indicates that an improvement in any one metric was followed by a similar improvement in all others. This consistent suggests that the models were approaching a performance barrier on the dataset itself rather than a model architecture compatibility.

The most important finding was the seemingly inherent ambiguity present in these yelp reviews. Even models specifically designed for Natural Language Processing (NLP), such as LSTM's, LLM's, and Transformers, were unable to breach the 70\% barrier. All this despite the dataset being large (700,000 text corpus), and well balanced (exactly 20\% per review category).

What this means is that reviews at 1 and 5 stars ("extreme experiences") were significantly easier to classify than reviews between 2 - 4 stars ("moderate experiences"). This is largely due to the fact that extreme experiences tend to contain clearer and more universally used sentiment signals, whereas moderate experiences are more nuanced and context-dependent.

A key reason for this is that moderate experiences are influenced by subjective user preferences that are not able to be expressed through a single body of text. For example, two users describing a similar level of customer service may assign different ratings (e.g 2 star vs. 3 star) depending on how they personally value service as compared to other factors (e.g quality of food).

Overall, the results suggest that the primary limitation in review categorization is not the models itself, but the inherent subjectivity of user experiences which result in an inescapable label noise present within the database.

\subsection{Limitations}
Several limitations affected the performance of the models.

First, the Yelp dataset introduces inherent label ambiguity, particularly in the 2 - 4 star range, as stated in the key findings. This limits maximum model performance regardless of architecture or complexity.

Second, most traditional models rely on the TF--IDF representation, which discards word order and removes a key piece of NLP and sentiment classification. This makes the traditional models ill-suited towards this task.

Finally, and arguably most importantly, model complexity was limited by computational resources and time. Some sacrifices to hyperparameters had to be made to ensure that models could be run on our domestic systems. This was especially true for deep learning models, which requires copious amounts of computation for its true results to shine through. Computation not easily found in consumer systems.

\subsection{Future Improvements}
Future work should focus on improving data representation and reducing label ambiguity rather than attempting to increase model complexity. One such improvement would be incorporating user metadata along with each review, such as user history, average score, or category. This may help to disambiguate moderate experiences where a single textual sentiment is insufficient to figure out user bias.

Further improvements can be made by switching from a classification to regression task. For example, classifying user sentiment as a continuous score rather than a discrete 1-5 class.

Finally, it should be noted that increased computational resources and time would allow for significantly more extensive hyperparameter tuning, larger or more complex models, and longer training schedules. This would particularly have an impact on our most complicated models, which were severely limited in their maximum length of review processing at only around 160 words. Which undoubtedly contributed to a lower score than the model should theoretically be capable of.

\subsection{Final Takeaway}
The project demonstrates that the primary challenge in the Yelp review classification is not the lack of powerful machine learning concepts, but rather the inherent subjectivity and ambiguity of the human experience itself.

While modern models are capable of extracting strong contextual signals from text, they are fundamentally limited when similar inputs correspond to different outputs due to personal experience and interpretation.

As a result, performance across all models trended towards a natural ceiling, particularly limited by the moderate experiences, as those contained significantly more domain overlap than the extreme experiences. Ultimately, this shows that in problems where labels are inherently subjective, having well-defined and consistent data is just as important as having a model sufficiently complex to process it.
